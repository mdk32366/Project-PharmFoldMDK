That last row is the point. `SKIP LOCKED` is not "untested" on SQLite — it is a **syntax
error** there. Turning that boundary into a *passing test* means the seam is now
specifically where `SKIP LOCKED` lives, not where the queue lives.

**Four amendments to D-009 §1**, each logged before the code and each pinned by a test
that goes red when the situation changes:

1. **`MAX_ATTEMPTS = 3`**, reasoned from measured facts rather than a round number: the
   host reliability floor is ~1 fatal per several weeks, so a job must survive one host
   loss; and a deterministic crasher costs a host crash per attempt, so the cap must be
   low. Three attempts = at most two crashes before the job stops asking. Reaped-out jobs
   terminate with a distinguishable `[reaped-out]` marker.
2. **Explicit `fail` is terminal, `attempts` preserved not zeroed.** The asymmetry is
   principled: an explicit fail means the worker *caught its own error* — usually
   deterministic, so retrying reproduces it. A stale reap means the worker *vanished* —
   usually environmental, so retrying is right. Absence is uninformative; a reported error
   is not.
3. **FIFO is contract, not inference.** §1 gave an index on `(status, created_at)`; an
   index makes an ordering *cheap*, not *guaranteed*. Now an explicit `ORDER BY`.
4. **FK deferral, with enforceable closure.** `jobs.analysis_id` is an unenforced integer
   until `protein_analyses` exists; `test_analysis_id_has_no_fk_yet` goes **red** the
   moment the constraint lands, so the gap cannot be silently outlived.

### D-015 — the project acquired a research question

This is the day's largest change. The deliverable was single-target analysis: enter a
protein, get a structure. It is now:

> **Does an ADC-suitability ranking built on structural features — computed from folds we
> run — differ from a ranking built on expression and evidence? Where they disagree, which
> disagreements are checkable against outcomes the world has already decided, and which are
> hypotheses?**

**Agreement is not the success condition.** A structural axis that merely reproduces an
expression-based ranking has added nothing.

- **Cohort:** Group A = the 82 prioritised targets of Kathad et al. 2024 (PLOS ONE, CC-BY);
  Group B = the 22 within A already tested as ADCs (the labelled set); Group C = TROP2,
  HER3, CLDN18.2 — targets their filters **excluded**, kept as out-of-cohort probes.
- **The comparator is a comparator, not an oracle.** Their 1–5 evidence score is built from
  literature / antibody-exists / protein-family / preclinical / clinical criteria — it
  substantially measures *how much attention a target has already received*. A target
  nobody has studied scores low by construction.
- **DL core moves up a level:** ESMFold stops being the deliverable and becomes the *input*
  to a small trained scorer. The network's output becomes a judgement that can be wrong.
- **§1a pre-registers disagreement** with three explanations — their blind spot (checkable),
  our new axis (hypothesis only), our pipeline is broken (a bug) — and states the honest
  prior: **for a first implementation, "we are wrong" is the most likely explanation for any
  given disagreement.** Four fixed diagnostics must rule it out before any claim.
- **Two negative outcomes named in advance**, including the one that would feel like
  success: if our structural score correlates *strongly* with their evidence score, that is
  **also null** — it means our features proxy for attention-and-precedent rather than
  measuring structure.

### The Postgres integration job — and the bug it caught on run one (D-017)

The job the whole seam had been pointing at. A throwaway Postgres 16 service container in
the gate, migrations applied by `alembic upgrade head` (**not** `create_all`).

**It earned its keep immediately, on its first run.** `env.py` ran the `search_path` SET
**before** `context.begin_transaction()`. In SQLAlchemy 2.0 that auto-opens a transaction
Alembic does not own — so `alembic upgrade head` logged *"Running upgrade → 0001_create_jobs"*,
**exited 0**, and the `CREATE TABLE` silently **rolled back**. Every production migration
would have no-op'd behind a green log.

> **This is the day's theme in its purest form: a true summary — "upgrade ran, exit 0" —
> hiding the disqualifying fact that nothing was committed.** The SQLite suite could not
> catch it, because SQLite lacks the transaction semantics that produced it. The JARVIS
> `create_all` gap was not a hypothetical to guard against; it was **live in our `env.py`**,
> and only real Postgres surfaced it.

Fixed (SET moved inside the committed transaction), with the finding and the run artefact
**recorded in `env.py` itself** so it cannot be reintroduced — the fix carries its own reason.

**Now proven on real Postgres 16 — `7 passed, 26 deselected`:**
- the Alembic chain builds the schema (not `create_all`)
- **`SKIP LOCKED` is atomic** — a row locked in one open transaction is invisible to another
  connection, which takes the next; all-locked yields `None`. *This is the assertion the
  entire D-012 seam existed to reach.*
- `complete` / `fail` / `reap_stale` and the attempts cap behave identically on real Postgres

**The green was verified, not assumed.** Code confirmed the 7 integration tests genuinely
*executed* rather than *skipped* — a service-container job that silently skips is a false
green, and a false green on the one job proving `SKIP LOCKED` would be worse than no job.

**D-017 — the job is deliberately NOT a required check yet.** A required service-container
job that flakes would deadlock every PR with no admin bypass — the D-013 hazard. Interim
safety: `deploy: needs postgres`, so a broken migration cannot deploy even if merged.
Promotion to required is an owner repo-settings action once the stated bar is met. **Two
consecutive green ticks so far.**

## Method notes — warnings for the next session, not trophies

**1. A summary is where the disqualifying detail goes to die.** Every reversal above was
already written down and was *true as stated*. `params_all_on_cuda=True` was true and missed
spill. "217 WHEA events" was true and missed severity. `pg_extension → 0 rows` was true and
missed absence. **`alembic upgrade head` exited 0 over an empty schema.** **When a check comes
back clean, ask what result would have looked identical but meant something worse.** If the
check cannot tell those apart, it did not do the work you think it did. Prefer the query that
could disqualify you: ask what is *available*, not what is *on*; ask for the breakdown, not the
total.

**1a. The corollary, earned three times today: a green is a claim too.** `7 passed, 26
deselected` was checked to confirm the integration tests **executed** rather than **skipped** —
a service-container job that silently skips is a false green on the one job proving
`SKIP LOCKED`. `38 passed, 8 skipped` was checked to confirm the GPU test **skipped cleanly
rather than erroring on import** — which would have presented identically and meant the module
was broken. **Confirm the thing you care about actually ran.** Exit codes and pass counts are
summaries.

**2. Make honesty structural, not documentary.** A caveat in a log entry gets skimmed. This
is the day's most transferable pattern:
`UnlockedFakeJobQueue` cannot be misread as coverage · `test_claim_is_unsupported_on_sqlite`
*passes*, so the boundary cannot be forgotten · `test_analysis_id_has_no_fk_yet` goes red the
moment the gap closes · the `env.py` transaction fix **carries its own reason in the file**, so
it cannot be reintroduced by someone who never saw the bug · `accelerate` is unpinned **and
flagged as unpinned**, rather than given an invented pin that would look complete · D-015 §1a
binds the **UI**, because a class-1 disagreement (checkable) and a class-2 disagreement
(hypothesis) look identical in a sorted table and mean entirely different things. **If the honesty depends on someone remembering, it will not survive.**

**3. Pre-register the negative outcome — especially when excited.** D-015 is the most
exciting entry in the log and therefore the one most likely to be filed on enthusiasm. Its
guard is §3's named negatives, written *before any result exists*, including the one that
would feel like success (high correlation with the comparator = our features proxy for
attention, not structure). This only works if it survives contact with a disappointing result.

**4. An identifier written from memory is a claim, not a lookup.** Accessions, versions,
commit SHAs, and cluster IDs are exactly the things that look authoritative and are cheap to
verify. Two of ten seed accessions were wrong; the one flagged as uncertain *was* wrong and
so was one that was not flagged. **Validate against an anchor before running anything at
scale** — the HER2 630 check is what turned a 20% error rate into a caught bug instead of a
dataset.
